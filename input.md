# input.md — Mars IoT Platform

## 1. Unified Internal Event Schema (MarsEvent)

All device data is normalised into the following JSON schema before being
placed on the `mars.normalized.events` Kafka topic.

```json
{
  "event_id":      "<uuid-v4>",
  "sensor_id":     "<string>",
  "schema_family": "<string>",
  "timestamp":     "<ISO-8601 UTC>",
  "value":         "<number | object>",
  "unit":          "<string | null>",
  "raw":           { "<original simulator payload>" }
}
```

### Field definitions

| Field | Type | Description |
|---|---|---|
| `event_id` | string (UUID v4) | Unique event identifier generated by the processing service |
| `sensor_id` | string | Matches the simulator's sensor name (e.g. `greenhouse_temperature`) |
| `schema_family` | string | Simulator schema family (e.g. `rest.scalar.v1`) |
| `timestamp` | string | ISO-8601 UTC; taken from the raw payload or generated at normalisation time |
| `value` | number \| object | Primary numeric reading; object for multi-field payloads |
| `unit` | string \| null | Standardized SI unit (e.g. `°C`, `%`, `µg/m³`) - values are converted to canonical units for consistency; null when not applicable |
| `raw` | object | Verbatim original payload from the simulator — never modified |

### Sensor → schema family mapping

| Sensor ID | Schema Family |
|---|---|
| `greenhouse_temperature` | `rest.scalar.v1` |
| `entrance_humidity` | `rest.scalar.v1` |
| `co2_hall` | `rest.scalar.v1` |
| `hydroponic_ph` | `rest.chemistry.v1` |
| `water_tank_level` | `rest.level.v1` |
| `corridor_pressure` | `rest.scalar.v1` |
| `air_quality_pm25` | `rest.particulate.v1` |
| `air_quality_voc` | `rest.chemistry.v1` |

## 2. Kafka Topic Contracts

| Topic | Producer | Consumers | Schema |
|---|---|---|---|
| `mars.raw.sensors` | ingestion-service | processing-service | `{sensor_id, raw_payload}` |
| `mars.normalized.events` | processing-service | state-service | MarsEvent (above) |
| `mars.actuator.commands` | processing-service | actuator-service | `{actuator_id, actuator_state, triggered_by_rule}` |

## 3. Automation Rule Schema (MongoDB)

```json
{
  "_id":            "<ObjectId>",
  "sensor_id":      "<string>",
  "operator":       "< | <= | = | >= | >",
  "threshold":      "<float>",
  "unit":           "<string | null>",
  "actuator_id":    "<string>",
  "actuator_state": "ON | OFF",
  "enabled":        "<boolean>",
  "description":    "<string | null>"
}
```

Example rule encoding:
> IF greenhouse_temperature > 28 °C THEN set cooling_fan to ON

```json
{
  "sensor_id": "greenhouse_temperature",
  "operator": ">",
  "threshold": 28.0,
  "unit": "°C",
  "actuator_id": "cooling_fan",
  "actuator_state": "ON",
  "enabled": true,
  "description": "Activate cooling fan when greenhouse is too hot"
}
```
